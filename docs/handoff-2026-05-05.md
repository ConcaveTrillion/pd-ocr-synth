## What's next: M02 — Recipe schema + validator

Spec: [`docs/specs/02-recipe-format.md`](specs/02-recipe-format.md).
Roadmap: [`docs/roadmap/02-recipe-schema.md`](roadmap/02-recipe-schema.md).

**Goal:** recipes load, validate, and surface errors with line-accurate
messages. After M02, users can iterate on `gaelic.yaml` even before
render works.

**Key deliverables (from the roadmap file):**

1. `pd_ocr_synth.recipe` — Pydantic v2 models for every block
   (Recipe, OutputBlock, CorpusEntry as discriminated union by `type`,
   TextTransform, Font, Rendering, Layout, DegradationStage,
   PublishBlock). YAML loader with `~` and `${ENV_VAR}` expansion,
   relative paths resolved against the recipe file's directory.
   Returns frozen Recipe objects.
2. `pd_ocr_synth.validation` — `validate_recipe(recipe) ->
   ValidationReport`. Surfaces missing files, unknown degradation
   `kind`s, conflicting `layout.mode` keys, unwritable destination,
   schema-version mismatch. Errors report file + line where possible.
3. CLI subcommands implemented (replacing stubs):
   - `list` — walks `$PD_OCR_SYNTH_RECIPES` + `./recipes/`
   - `validate <recipe>` — exit 0/3
   - `describe <recipe>` — resolved config + corpus stats placeholder
   - `init <name>` — scaffold `recipes/<name>/recipe.yaml`
4. JSON Schema export — `pd-ocr-synth schema` (or Make target) emits
   `docs/specs/recipe.schema.json` from the pydantic models for VS
   Code's YAML extension
5. Tests covering each rule, path resolution edge cases, and the
   range/scalar/weighted-choice union pattern

**Validation criterion:** `pd-ocr-synth validate gaelic` exits 0
against the bundled recipe; `recipes/gaelic.yaml` opens cleanly in VS
Code with the YAML extension and gets schema completion.

**Probable runtime deps to add to `pyproject.toml`** (currently empty
`dependencies = []`): `pydantic>=2.0`, `pyyaml`, `ruamel.yaml`
(optional, for line-accurate errors).

## Architecture decisions worth knowing

These are choices already made — don't relitigate without reason.

1. **Concrete-first, not framework-first.** Build the Gaelic path as
   straightforward code, then extract abstractions when the second
   recipe (Fraktur, etc.) forces them.
2. **Recipe YAML is the contract.** No separate config file format;
   the recipe is the single declarative unit.
3. **CLI is the contract too.** A NiceGUI preview surface is M11 and
   intentionally read-only on recipes (only "Diff & save" writes).
   Don't add a YAML editor — VS Code with the JSON Schema is the
   editor.
4. **Local-first output, optional HF publish.** Render writes the
   `pd-ocr-trainer/v1` layout to local disk; `publish` is a separate
   subcommand that ships to HF Datasets. See
   `docs/specs/10-publishing.md` and workspace `DATASETS.md`.
5. **Free fonts only, never bundled.** The fetch script downloads from
   upstream sources; no fonts in the repo. Gaelchló is permitted on
   two grounds (Celtic-language + non-commercial).
6. **FontForge / Gaedhilge is fully optional.** Default `fetch-fonts`
   does NOT install FontForge or download Gaedhilge. The opt-in
   `--with-gaedhilge` flag downloads the `.sfd` and only compiles it
   if `fontforge` is already on PATH. **Do NOT add auto-install of
   FontForge.** This was explicitly scoped down — earlier exploration
   built an AppImage installer with apt fallback and the user
   reverted it.
7. **`make setup` vs `make install`.** Setup = dev env (uv sync +
   pre-commit). Install = `uv tool install --reinstall .` for
   end-users. Mirrors `pd-ocr-labeler`. Don't put dev-env install
   logic into the `install` target.
8. **No `dev-env-setup.sh`.** Removed deliberately — `make setup` is
   the single source of truth. The peer `pd-ocr-trainer/dev-env-setup.sh`
   is a legacy pip-based artifact.
9. **MVVM + layered.** When implementation starts, mirror the
   `pd-ocr-labeler` pattern: state → operations → viewmodels → views.
   Applies to both the engine (state→operations is enough; no UI)
   and the eventual M11 preview UI.
10. **No backwards-compat shims.** This is a greenfield project; no
    legacy code paths to preserve. Don't add feature flags or "future
    options" beyond what a milestone needs.
