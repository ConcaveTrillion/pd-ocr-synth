# 10 — Publishing to Hugging Face

This spec describes how `pd-ocr-synth` publishes a rendered dataset to a
Hugging Face Datasets repo. It implements the producer half of the
workspace-wide contract in
[`/workspaces/ocr-container/DATASETS.md`](../../../DATASETS.md). Read that
first; this doc covers only the synth-specific details.

## When to publish

Publish is **explicit and separate from render**. Render writes the local
`pd-ocr-trainer/v1` layout for fast trainer iteration; publish takes that
output and ships it to HF for sharing or cloud training.

```bash
pd-ocr-synth render gaelic           # local; fast iteration
pd-ocr-synth publish gaelic          # one-shot upload to HF
```

Publishing requires that a render has already happened — it does not
re-render. Pass `--render-first` to chain them.

## What gets uploaded

`publish` reads `<destination>` from the recipe's `output:` block and
emits one HF dataset repo containing:

| Path | Source | Notes |
|------|--------|-------|
| `data/*.png` | `<destination>/images/` | Recognition mode |
| `data/*.parquet` | Built from local images + `pages.json` | Detection mode |
| `metadata.jsonl` | Built from local `labels.json` + `manifest.jsonl` | Recognition only |
| `recipe.snapshot.yaml` | Copied from local output | Provenance |
| `README.md` | Generated; see "Dataset card" | Auto-generated |

Local-only files NOT uploaded: `manifest.jsonl` (rolled into
`metadata.jsonl`), `labels.json` (same), `stats.json` (rolled into the
README).

## Format conversion — recognition

Local layout:
```
images/0000000.png + labels.json entry {"0000000.png": "Séadna"} + manifest.jsonl record
```
becomes one row in HF `metadata.jsonl`:
```json
{"file_name": "data/0000000.png", "text": "Séadna",
 "font": "bungc/bungc.otf", "font_size_pt": 14.0,
 "degradations": ["skew", "paper_texture", "jpeg"],
 "corpus": "wikisource:mul:Séadna"}
```

The `file_name` column is the HF imagefolder convention. All other
columns are flat strings/numbers/lists; nothing nested.

## Format conversion — detection

Detection mode reads `pages.json` + the image files and writes parquet
shards via `datasets.Dataset.from_generator(...).push_to_hub(...)`.

Schema:
```python
{
  "image": Image(),
  "size": Sequence(Value("int32")),                # [width, height]
  "lines": Sequence({
    "bbox": Sequence(Value("int32"), length=4),
    "text": Value("string"),
    "words": Sequence({
      "bbox": Sequence(Value("int32"), length=4),
      "text": Value("string"),
    }),
  }),
  "font": Value("string"),
  "degradations": Sequence(Value("string")),
}
```

Shard size: 500 MB target per parquet file. The `datasets` library
handles splitting automatically.

## Recipe `publish:` block

Optional. Holds defaults so `publish` runs with no args.

```yaml
publish:
  hf_dataset:
    repo: ntw8532/pd-ocr-synth-gaelic
    private: false
    license: cc-by-4.0
    tags: [ocr, gaelic, irish, pd-ocr, synthetic]
    language: [ga]
    description_file: ./gaelic/README.md.template   # optional override
```

CLI flags (`--repo`, `--private`, `--license`, etc.) override recipe
values when both are present.

## Idempotency

Publish is **idempotent against repo state**:

1. Compute a content SHA over the local output (image bytes +
   metadata + recipe snapshot).
2. Read the latest commit's `card_data.pd-ocr-content-sha`.
3. If equal → exit 0 with "no changes" and do not commit.

This makes `pd-ocr-synth publish gaelic` safe to invoke from CI on
every run; only an actual recipe or output change triggers a new
commit.

## Versioning

Each publish creates one commit. Tag it explicitly when you want a
pin:

```bash
pd-ocr-synth publish gaelic --tag v2026.05.05
```

The trainer's profile config can reference the tag for a stable pin:

```yaml
sources:
  - hf://ntw8532/pd-ocr-synth-gaelic@v2026.05.05
```

`--message "..."` overrides the auto-generated commit message
(default: `pd-ocr-synth render @<recipe-sha>`).

## Authentication

Uses `huggingface_hub`'s default auth resolution:

1. `--token <token>` flag (least preferred; ends up in shell history)
2. `HF_TOKEN` env var (recommended)
3. `~/.cache/huggingface/token` (from `hf auth login`)

Public dataset writes need a token with `write` scope on the target
namespace. Read-only operations during render do not need a token.

## Tooling used

| Library / call | Purpose |
|---|---|
| `huggingface_hub.HfApi.create_repo` | First-time repo creation |
| `huggingface_hub.HfApi.upload_large_folder` | The actual upload (chunks into commits, resumes on failure) — used for recognition |
| `datasets.Dataset.from_generator(...).push_to_hub(...)` | Used for detection (parquet sharding) |
| `huggingface_hub.HfApi.list_repo_commits` | Idempotency check |
| `huggingface_hub.HfApi.create_tag` | `--tag` flag |

No custom upload code; all primitives come from the official libraries.

## Errors and recovery

| Failure | Behavior |
|---------|----------|
| Network error mid-upload | `upload_large_folder` resumes on retry; partial commit ok |
| Auth error | Exit code 7; print the auth resolution chain |
| Repo doesn't exist | Auto-create unless `--no-create` |
| Missing local render | Exit code 5 with "render first or use --render-first" |
| Local output corrupt (manifest mismatch) | Exit code 6; refuse to upload partial data |

## CLI summary

```bash
pd-ocr-synth publish <recipe>
  [--repo OWNER/NAME]      # default from recipe.publish.hf_dataset.repo
  [--private | --public]   # default from recipe; else public
  [--license LICENSE]
  [--tag VERSION]
  [--message MESSAGE]
  [--token TOKEN]
  [--render-first]         # render before publishing
  [--no-create]            # fail instead of auto-creating the repo
  [--dry-run]
```
